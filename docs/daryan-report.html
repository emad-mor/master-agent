Centerpiece

How the layered memory works

Daryan doesn't keep a flat transcript. Memory is four tiers that age downward: a turn starts verbatim, becomes a one-paragraph summary, then folds into a long-term theme. Core is the only global tier; the rest are per project, so juggling many projects never bleeds one conversation into another. Each project also keeps its own Claude session id, so switching projects resumes the right context.

On every Send

Daryan builds the prompt's memory block as core + themes + summaries + recent, injects it (on the first turn, after an idle gap, or when the prompt cache has expired — otherwise it leans on --resume), then runs eviction so the tiers stay within their limits.

✂️

Optimization #1 — lean injection

The verbatim recent block was ~80% of per-turn weight. Now each recent turn is compacted at injection time (stored turns stay full, so on-reload history is intact): UI markers stripped, big code blocks truncated to a reference, length capped, prompts clipped.

♻️

Optimization #2 — free eviction

The voice-first reply already carries a [[SUMMARY]] recap — a ready-made turn summary. Eviction now reuses it as the mid-tier summary instead of spawning a Haiku call. Only turns without one fall back to the model.

Known trade-offs

Honest limits

  • Word-level voice highlight is time-interpolated (Kokoro emits no word timestamps) — it can drift; sentence-level is exact.
  • Voice-first home replies collapse to the summary once a stream finishes (full text shows while streaming).
  • Injected memory truncates code blocks — the agent re-reads files when it needs verbatim code.
  • The in-app agent edits real files via the CLI (no sandbox); the “Daryan source” scope rule guards against accidental self-edits.
